1 # F´Continuous Integration
2 
3 F´continuous integration was developed to work with Jenkins. The continuous integration project is
4 designed to checkout a new pull request that has been submitted to F´, merge it with the base
5 branch, and then build the merged construct. Then the unit tests are built and run against this
6 merge. This proves that the build is ready for review and merge.
7 
8 ## Jenkins Description
9 
10 The Jenkins setup exists in the `config.xml` file in this directory. It should be noted that the
11 secrets (passwords, keys, etc.) have been stripped and should be replaced.
12 
13 The Jenkins CI is configured with the expressed purpose of checking the build and unit tests for
14 a given pull request. It does not build on a set interval, nor does it build a specific branch
15 (master, devel) but rather whatever branch the pull request was based on.
16 
17 The Jenkins CI is setup to be built inside a docker container. This docker container is used to
18 encapsulate the exact build environment without needing to setup a specific Jenkins machine with
19 this build environment. It needs to be run on a Docker capable Jenkins machine. The Docker setup
20 that powers this is at: [mk/docker](../docker/).
21 
22 Once Jenkins CI builds the Docker container from the Dockerfile, it runs the Python Test Framework,
23 [ptf](/ptf), which includes the following test suites:
24 
25  1. Reference Build and Unit Tests
26  2. Autocoder Unit Tests

28 ## Using Jenkins
29 
30 In order to use the existing Jenkins configuration, one can post it to the server of their choice
31 using the following basic command. This command may need to be altered to supply credentials or
32 setup proxies, but this is outside this guide.
33 
34 This `config.xml` contains `FILL-ME` tokens where various project specific configuration items
35 belong. These mostly represent passwords, or other secrets that should not be exposed. A user can
36 edit the config.xml file directly before POSTing the XML to the server. Alternatively, the user can
37 POST the XML and make the edits in the Jenkins GUI.
38 
39 
40 **POST Jenkins Config as New Job**
41 ```bash
42 fprime>curl -X POST http://<jenkins host>/createItem?name=<job name>' --header "Content-Type: application/xml" -d mk/ci/config.xml
43 ```
44 Here the user must supply the `<jenkins host>` and `<job name>` to create a job on the given host with the given job name.

48 ### Daily and Manual Builds
49 
50 Some users like to have an repeating build of Jenkins on a Daily or Weekly schedule to ensure that
51 the common branch (master, devel) are consistent and correct. This can be done by setting a
52 scheduled build trigger and, if using the above `config.xml`, the `${ghprbActualCommit}` and
53 `${ghprbTargetBranch}` environment variables should be set to the desired branch to build. This can be done
54 in Jenkins's environment setup. This allows manual and scheduled builds without creating a separate
55 job by defaulting the variables used by the Pull Request builder. The user could also setup a
56 separate job to handle this.
